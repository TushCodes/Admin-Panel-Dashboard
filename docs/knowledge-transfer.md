# Knowledge Transfer Guide

This guide turns the Admin Panel Dashboard into a structured onboarding ground for junior developers and frontend-focused developers who need to become productive across the full project.

## Learning goals

By the end of this KT path, a new developer should be able to:

- Explain how a request moves from the browser to Express routes, controllers, services, database helpers, and back to the UI.
- Run the project locally, execute the automated checks, and understand which checks require a live database.
- Safely add or modify an API endpoint using the route/controller/service/database boundaries already in the project.
- Update the static Vue frontend without introducing a separate build pipeline.
- Make Prisma schema changes, generate the client, and reason about migration files.
- Debug common failures using API response shapes, error helpers, and the Node.js test runner.

## Audience assumptions

This curriculum assumes the developer can already read basic JavaScript, HTML, and CSS. It does not assume prior Express, Prisma, database migration, or backend testing experience. Frontend-leaning developers should pay extra attention to the backend request lifecycle and database sections.

## Recommended KT sequence

### 1. First run and project map

1. Install dependencies with `npm install`.
2. Generate Prisma Client with `npm run prisma:generate` after dependency installation or schema edits.
3. Run `npm test` to verify the database-free test suite.
4. Start the app with `npm start` and visit `/`, `/health`, `/auth/login`, and `/admin`.
5. Read `README.md` for the high-level runtime, routes, and folder map.

Key files to open first:

- `server.js` - app creation, middleware, static frontend hosting, route registration, health checks, and error handling.
- `routes/index.js` - resource route mounting.
- `controllers/index.js` - controller factory exports.
- `services/dataAggregator.js` - example of business logic that merges database and external API records.
- `models/schema.prisma` - database model source of truth.
- `frontend/app.js` - the lightweight Vue shell and route switching.

### 2. Request lifecycle walkthrough

Use `GET /consignments/aggregated-consignments` as the first guided example:

1. `server.js` creates the Express app and calls `registerRoutes(app)`.
2. `routes/index.js` mounts consignment routes under `/consignments`.
3. `routes/consignments.js` maps `GET /aggregated-consignments` to the consignment controller.
4. `controllers/consignments.js` returns a JSON API response with aggregated consignment data.
5. `services/dataAggregator.js` pulls records from database and external API sources, validates external records, and tags each record with its source.
6. `db/consignments.js` contains the Prisma-backed data access functions.
7. `utils/asyncHandler.js` and `utils/errorHandling.js` standardize async error behavior and error responses.

Exercise: add a temporary `console.log` at one layer at a time, call the endpoint, and remove the log before committing.

### 3. Backend boundaries

Keep backend changes within these boundaries:

- `routes/` should only define HTTP paths, verbs, URL params, and middleware wiring.
- `controllers/` should translate HTTP requests into application operations and HTTP responses.
- `services/` should contain business rules that combine or validate data across sources.
- `db/` should contain Prisma/database-specific access logic.
- `utils/` should contain reusable helpers that are not tied to one resource.

Practice task: add a read-only endpoint for one small computed value, write a unit test around the service/controller behavior, then remove or keep it only if it is part of the assigned story.

### 4. Frontend boundaries

The current frontend is intentionally simple: static files plus Vue loaded from the server-hosted dependency bundle. There is no Vite, React, TypeScript, or frontend router.

- `frontend/index.html` hosts the app root and static asset references.
- `frontend/app.js` initializes Vue and switches between known pages based on `window.location.pathname`.
- `frontend/routes.js` stores frontend route metadata.
- `frontend/pages/` contains page-level components.
- `frontend/components/` contains reusable visual components.
- `frontend/tailwind.css` is the Tailwind input file.
- `frontend/assets/tailwind.css` is generated by `npm run build:frontend` / `npm run build:css`.

Practice task: add a small piece of route metadata and display it on an admin page without changing backend routes.

### 5. Database and Prisma workflow

Prisma work should follow this order:

1. Edit `models/schema.prisma`.
2. Run `npm run prisma:generate`.
3. Create a migration with `npm run prisma:migrate:dev` when a real schema change is intended.
4. Review generated files and migration SQL before committing.
5. Run database-backed checks only when `DATABASE_URL` points to a safe development database.

Do not commit secrets. Use `.env` locally and keep real credentials out of source control.

### 6. Testing expectations

Use the Node.js test runner as the default quality gate:

- Run `npm test` for database-free unit and integration coverage.
- Run `npm run build:frontend` after frontend styling changes.
- Run `npm run test:db` only when a valid development `DATABASE_URL` is configured.

Good first tests for new developers:

- A utility helper test in `test/test_utils.js`.
- A controller test with injected fake dependencies.
- A service test for record validation or aggregation behavior.
- A server route smoke test in `test/test_server.js`.

### 7. Debugging checklist

When something fails, ask these questions in order:

1. Is the failing path frontend-only, API-only, database-only, or cross-layer?
2. Does the endpoint return the standard `{ success, data }` or `{ success, error }` shape?
3. Is the error thrown inside an async route wrapped with `asyncHandler`?
4. Does the failing data shape match `models/schema.prisma` and `CONSIGNMENT_RECORD_FIELDS` expectations?
5. Is `DATABASE_URL` required for this path, and is it configured for a safe local database?
6. Did generated Prisma files or generated CSS change, and are those changes intentional?

### 8. Definition of ready for project work

A developer is ready to take a small production ticket when they can independently:

- Start the app and run the default test suite.
- Trace one endpoint from frontend route to database helper.
- Explain where to put route, controller, service, database, utility, and frontend changes.
- Add or update at least one test for their change.
- Describe whether their change requires Prisma generation, a migration, frontend CSS generation, or database credentials.

## Suggested mentor-led sessions

| Session | Topic | Outcome |
| --- | --- | --- |
| 1 | Local setup and project map | Developer can run the app and name every top-level folder's purpose. |
| 2 | Express request lifecycle | Developer can trace `/consignments/aggregated-consignments`. |
| 3 | Data model and Prisma | Developer can explain schema, client generation, and migration safety. |
| 4 | Static Vue frontend | Developer can make a small UI change and rebuild CSS. |
| 5 | Testing and debugging | Developer can add a focused test and diagnose a failing endpoint. |
| 6 | Capstone ticket | Developer completes a small vertical slice with review feedback. |

## Capstone exercises

Choose one exercise based on the developer's starting point:

1. **Frontend-focused:** add an empty-state message to one admin page and cover any data-shaping helper with a test.
2. **Backend-focused:** add a small validation rule in a service and test success and failure cases.
3. **Full-stack:** add a read-only computed field to an API response and display it on an admin page.

For every capstone, the developer should open a PR that includes a short testing note and a short explanation of which layers changed.
